Этап 1
## Этап 1: Stub + Description + Валидация

### Входные данные задачи

{ВСТАВЬ СЮДА ОПИСАНИЕ ФАЗЫ/ЗАДАЧИ}

---

### Инструкции

1. **Stub-код:** Создай код-заглушки (stub) для всех классов, методов и функций перечисленных в задаче выше. Каждый метод должен содержать `raise NotImplementedError` или `pass`, сигнатуры с типизацией и docstring с кратким описанием назначения.

2. **Description-файл:** В папке `Description/` создай файл с именем `description_{название_модуля}.md`. В нём для **каждого** класса, метода и функции из stub опиши:
   - Назначение (что делает)
   - Входные параметры (имя, тип, смысл)
   - Выходные данные (тип, смысл)
   - Ключевую логику / алгоритм (кратко, но достаточно для написания тестов)
   - Граничные случаи и ожидаемые ошибки
   - Связи с другими компонентами (зависимости внутри модуля и внешние)

3. **Inline-ссылки:** В каждом stub-файле над классом/функцией добавь комментарий-ссылку на соответствующий раздел description-файла. Формат:
```python
   # Description: Description/description_{название_модуля}.md#название-секции
```

4. **🔍 Валидация соответствия (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   После завершения шагов 1–3 проведи финальную проверку. Построй таблицу валидации по следующей структуре:

   | № | Компонент из задачи | Stub создан? | Description есть? | Description достаточен для написания тестов? | Расхождения / Замечания |
   |---|--------------------|--------------|--------------------|----------------------------------------------|------------------------|

   **Правила проверки:**
   - Пройди по КАЖДОМУ компоненту/классу/методу из исходной задачи и убедись что для него есть stub И description.
   - Проверь что description содержит достаточно информации для написания TDD-тестов на Этапе 2: входы, выходы, поведение, граничные случаи.
   - Проверь что description НЕ противоречит задаче (не добавлено лишнего функционала, не потеряна заявленная функциональность).
   - Проверь что все зависимости между компонентами отражены в description.
   - Если найдены расхождения — **исправь их** перед завершением этапа и отметь что было исправлено.

   **Итоговый вердикт:** В конце таблицы напиши один из статусов:
   - ✅ `PASS` — все компоненты покрыты, descriptions достаточны для TDD
   - ⚠️ `PASS WITH NOTES` — покрыто, но есть замечания (перечисли)
   - ❌ `FAIL` — есть пропуски или критические расхождения (исправь и повтори валидацию)


## Этап 2: TDD-тесты + Валидация

### Входные данные

- **Задача:** {НАЗВАНИЕ ФАЗЫ / ЗАДАЧИ}
- **Stub-файл(ы):** {ПУТИ К STUB-ФАЙЛАМ СОЗДАННЫМ НА ЭТАПЕ 1}
- **Description-файл:** {ПУТЬ К DESCRIPTION-ФАЙЛУ}

---

### Инструкции

1. **Изучи описания:** Перед написанием тестов внимательно прочитай description-файл целиком. Для каждого класса/метода/функции выдели:
   - Ожидаемое поведение (что должно происходить при нормальном вызове)
   - Граничные случаи (пустые входы, нулевые значения, экстремальные параметры)
   - Ожидаемые ошибки (какие исключения и при каких условиях)
   - Зависимости и взаимодействия с другими компонентами

2. **Напиши TDD-тесты:** В соответствующем тестовом файле создай тесты используя pytest. Структура тестов:
```
   Для каждого класса/метода из description:
   ├── test_{method}_basic          — базовый happy-path
   ├── test_{method}_edge_cases     — граничные случаи
   ├── test_{method}_error_handling — невалидные входы, исключения
   └── test_{method}_integration    — взаимодействие с другими компонентами (если описано)
```

   **Требования к тестам:**
   - Каждый тест должен иметь docstring объясняющий **что** проверяется и **почему**
   - Тесты должны быть независимы друг от друга
   - Используй `pytest.raises` для проверки исключений
   - Используй `pytest.approx` для числовых сравнений с плавающей точкой
   - Используй фикстуры (`@pytest.fixture`) для общих тестовых данных
   - Каждый тест должен проверять ОДНО конкретное поведение

3. **Inline-ссылки на описание:** Над каждой группой тестов добавь комментарий-ссылку на соответствующий раздел description-файла:
```python
   # Тесты по: {путь_к_description}#название-секции
   class TestНазвание:
       ...
```

4. **🔍 Валидация покрытия (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   После написания всех тестов проведи проверку полноты. Построй таблицу:

   | № | Класс/Метод из Description | Тест basic | Тест edge | Тест error | Тест integration | Пропущенное поведение из description |
   |---|---------------------------|------------|-----------|------------|------------------|--------------------------------------|

   **Правила проверки:**
   - Пройди по **каждому** пункту из description-файла и убедись что для него есть хотя бы один тест.
   - Проверь что **все граничные случаи** описанные в description покрыты тестами.
   - Проверь что **все ожидаемые ошибки** из description имеют соответствующий `test_*_error_handling`.
   - Проверь что тесты проверяют **именно то поведение** что описано, а не что-то другое или выдуманное.
   - Проверь что тесты **не тестируют внутреннюю реализацию** — только публичный контракт (входы → выходы / побочные эффекты).
   - Убедись что stub-файл импортируется в тестах корректно и все имена классов/методов совпадают.

5. **🔍 Обратная проверка description (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   Если при написании тестов обнаружились **пробелы в description** (неясное поведение, отсутствующие граничные случаи, неопределённые типы), то:
   - Перечисли найденные пробелы
   - Дополни description-файл недостающей информацией
   - Отметь что было дополнено

   **Итоговый вердикт:**
   - ✅ `PASS` — каждый пункт description покрыт тестами, пробелов нет
   - ⚠️ `PASS WITH NOTES` — покрыто, но description был дополнен (перечисли дополнения)
   - ❌ `FAIL` — есть непокрытые пункты (допиши тесты и повтори валидацию)

## Этап 3: Реализация + Валидация

### Входные данные

- **Задача:** {НАЗВАНИЕ ФАЗЫ / ЗАДАЧИ}
- **Stub-файл(ы):** {ПУТИ К STUB-ФАЙЛАМ}
- **Description-файл:** {ПУТЬ К DESCRIPTION-ФАЙЛУ}
- **Тесты:** {ПУТИ К ТЕСТОВЫМ ФАЙЛАМ}
- **План:** `RegenTwin_Update_Implemention_Plan.md`
- **Репозиторий:** https://github.com/RageBober/RegenTwin

---

### Инструкции

1. **Реализация по тестам:** На основании TDD-тестов реализуй полный функционал. Порядок работы:
   - Бери один класс/метод за раз
   - Реализуй минимальный код чтобы тесты для этого метода прошли
   - Убедись что тесты проходят перед переходом к следующему методу
   - Не пиши код который не проверяется тестами

2. **Итеративное исправление:** Если код не проходит тесты:
   - Прочитай сообщение об ошибке
   - Определи причину: баг в реализации или некорректный тест
   - Если баг в реализации — исправь код
   - Если тест некорректен (противоречит description) — исправь тест И задокументируй причину исправления
   - Повтори запуск тестов

3. **🔍 Проверка полноты реализации (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   После того как все тесты проходят, построй таблицу:

   | № | Тест | Статус (pass/fail) | Если fail — причина и действие |
   |---|------|--------------------|-------------------------------|

   **Требование:** Все строки должны быть `pass`. Если есть `fail` — вернись к шагу 2.

4. **Линтеры и проверки качества (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   Запусти последовательно и исправь все найденные проблемы:
```bash
   # 1. Тесты
   pytest {путь_к_тестам} -v

   # 2. Форматирование
   black {все_файлы_реализации_и_тестов}

   # 3. Линтер
   ruff check {все_файлы_реализации_и_тестов} --fix

   # 4. Типизация
   mypy {файлы_реализации}
```

   Если после `black` или `ruff --fix` тесты перестали проходить — повтори pytest. Цикл продолжается пока **все четыре проверки** не пройдут чисто.

   Результат зафиксируй:

   | Проверка | Результат | Ошибки найдены? | Исправлено? |
   |----------|-----------|-----------------|-------------|
   | pytest   |           |                 |             |
   | black    |           |                 |             |
   | ruff     |           |                 |             |
   | mypy     |           |                 |             |

5. **🔍 Валидация соответствия описанию (ОБЯЗАТЕЛЬНЫЙ ШАГ):**

   Открой description-файл и пройди по **каждому** пункту:

   | № | Поведение из Description | Реализовано? | Тест проходит? | Расхождения |
   |---|--------------------------|--------------|----------------|-------------|

   **Правила проверки:**
   - Для каждого описанного поведения убедись что код **действительно** его реализует, а не просто проходит тест формально.
   - Проверь что реализация **не добавляет** поведения которого нет в description (scope creep).
   - Проверь что **граничные случаи** из description обрабатываются корректно.
   - Проверь что **зависимости и взаимодействия** между компонентами работают как описано.
   - Если найдены расхождения — **исправь реализацию**, перезапусти тесты и линтеры.

6. **Обнови план:** Обнови `RegenTwin_Update_Implemention_Plan.md`:
   - Отметь фазу как завершённую (или частично с указанием %)
   - Перечисли реализованные компоненты
   - Отметь если что-то отложено или изменено

7. **Коммит в GitHub:**
```bash
   git add {все_файлы_над_которыми_велась_работа}
   git commit -m "{Фаза X.X}: {Краткое описание}

   Реализовано:
   - {конкретный список компонентов}

   Тесты: все проходят (pytest)
   Качество: black, ruff, mypy — чисто"

   git push origin main
```

   **Требования к коммиту:**
   - В описании перечисли **конкретно** что реализовано — не общие фразы
   - Добавь **только** файлы над которыми велась работа в этой фазе
   - Если description-файл был изменён на этом этапе — включи и его

   **Итоговый вердикт:**
   - ✅ `PASS` — все тесты проходят, линтеры чисты, реализация соответствует description, коммит сделан
   - ⚠️ `PASS WITH NOTES` — всё работает, но были отклонения от description (перечисли)
   - ❌ `FAIL` — есть непройденные тесты или расхождения с description (исправь и повтори)
